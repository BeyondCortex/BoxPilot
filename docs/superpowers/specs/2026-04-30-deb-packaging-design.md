# BoxPilot Plan #9 — `.deb` Packaging, Install Flow, GUI Auto-launch

**Date:** 2026-04-30
**Spec parent:** `docs/superpowers/specs/2026-04-27-boxpilot-linux-design.md` §15
**Status:** Design (pre-plan)

## 1. Scope

Plan #9 turns the dev-only `make install-helper` flow into a real Debian
package. After plan #9 a Debian / Ubuntu user can run

```bash
sudo apt install ./boxpilot_0.1.0_amd64.deb
```

and get a working install with:

- the privileged helper `boxpilotd` registered for D-Bus + systemd activation,
- the unprivileged Tauri GUI launchable from the desktop menu (and from
  `boxpilot` on the CLI),
- D-Bus, polkit, systemd, desktop and icon files in the canonical FHS
  locations,
- correct upgrade and removal semantics (user data is preserved on remove,
  cleared on purge).

In scope:

- A complete `debian/` source tree (`debhelper-compat = 13`, `dh` sequencer)
  driving `cargo` for both the helper and the Tauri Rust side, and `npm` for
  the Vite frontend.
- A new shipped systemd unit `boxpilotd.service` that D-Bus activates via
  `SystemdService=`. Replaces the TODO comment in
  `app.boxpilot.Helper.service`.
- A `app.boxpilot.desktop` desktop entry plus a hicolor icon set
  (32 / 128 / 256 px).
- `postinst` / `prerm` / `postrm` maintainer scripts that:
  - reload dbus and polkit,
  - update GTK icon and desktop databases,
  - on `remove`: leave user data alone,
  - on `purge`: remove `/etc/boxpilot/`, `/var/lib/boxpilot/`,
    `/var/cache/boxpilot/`, and the runtime-generated
    `boxpilot-sing-box.service` unit,
  - never touch a user-owned external `sing-box.service`.
- A `make deb` orchestration target plus a contributor doc paragraph that
  explains how to build the package locally.
- README rewrite: the user-facing quick start becomes the `apt install`
  flow; the existing `make install-helper` block is preserved under a
  "Building from source / dev" section.

Out of scope (not blocking v1.0):

- `.rpm` / Arch / nix packaging — Debian / Ubuntu are the v1.0 reference
  target per spec §1.
- Signed releases, GPG-signed `.deb`s, or apt repositories. The first
  release ships the `.deb` as a GitHub release artifact; signing is a
  follow-up.
- Reproducible builds. We do not pin Cargo / npm fetches, so a local rebuild
  is the user's responsibility.
- Auto-launch on login. The desktop entry is enough for the v1.0 user
  story ("find BoxPilot in the menu, click it"). Login auto-start is a
  user-toggleable preference, not a packaging concern.
- A non-`github.com` core mirror knob (mentioned as deferred work in plan
  #2 §554). The release URL stays hardcoded; the knob is a future plan.
- Cross-arch builds. The `.deb` is built on the host architecture; the
  control file declares `Architecture: any` so a future `arm64` build just
  needs another build host.
- `--no-audit --no-fund` style npm tweaks beyond what we already need to
  get a clean offline-ish build.

## 3. File layout (installed by the package)

```
/usr/bin/boxpilot                                                      # GUI binary symlink → /usr/lib/boxpilot/boxpilot
/usr/lib/boxpilot/boxpilot                                             # Tauri 2 GUI binary
/usr/lib/boxpilot/boxpilotd                                            # privileged helper
/usr/lib/systemd/system/boxpilotd.service                              # D-Bus systemd activation unit
/usr/share/dbus-1/system-services/app.boxpilot.Helper.service          # D-Bus service file (with SystemdService=)
/usr/share/dbus-1/system.d/app.boxpilot.helper.conf                    # D-Bus permissions
/usr/share/polkit-1/actions/app.boxpilot.helper.policy                 # polkit actions
/usr/share/polkit-1/rules.d/49-boxpilot.rules                          # polkit rules
/usr/share/applications/app.boxpilot.desktop                           # desktop entry
/usr/share/icons/hicolor/32x32/apps/boxpilot.png
/usr/share/icons/hicolor/128x128/apps/boxpilot.png
/usr/share/icons/hicolor/256x256/apps/boxpilot.png
/usr/share/doc/boxpilot/copyright                                      # GPL-3.0
/usr/share/doc/boxpilot/changelog.Debian.gz                            # debhelper-managed
/usr/share/doc/boxpilot/README.md                                      # informational
/etc/boxpilot/                                                         # empty conffile dir; helper writes config in here
/var/lib/boxpilot/                                                     # empty; helper writes core releases here
/var/cache/boxpilot/                                                   # empty; helper writes diagnostics here
```

Notes:

- `/usr/bin/boxpilot` is a relative symlink (`../lib/boxpilot/boxpilot`). The
  GUI binary lives under `/usr/lib/boxpilot/` so the helper and GUI sit
  side-by-side; this matches the directory the polkit rules file already
  treats as the package's home and lines up with the existing
  `Exec=/usr/lib/boxpilot/boxpilotd` line in the D-Bus service file.
- `/etc/boxpilot/`, `/var/lib/boxpilot/`, `/var/cache/boxpilot/` are
  declared in `debian/boxpilot.dirs`. They are **not** marked as conffiles
  and have no shipped contents, so dpkg does not prompt about user
  modifications.

`debian/source/format` is `3.0 (native)` because BoxPilot's git tree itself
is the upstream — there is no separate upstream tarball, and we do not
generate quilt patches against external sources.

## 5. Build pipeline (`debian/rules`)

`debian/rules` uses the `dh` sequencer with explicit overrides because
`dh_auto_*` does not understand a Cargo + npm workspace.

```makefile
#!/usr/bin/make -f
export DH_VERBOSE = 1
export CARGO_HOME = $(CURDIR)/debian/cargo-home
export CARGO_TARGET_DIR = $(CURDIR)/target

# Avoid network during package build *only* if the user has pre-fetched
# (CI/release path). For now both cargo + npm are allowed to fetch.

%:
	dh $@

override_dh_auto_clean:
	cargo clean || true
	rm -rf debian/cargo-home frontend/dist frontend/node_modules

override_dh_auto_build:
	cargo build --release --locked -p boxpilotd
	cd frontend && npm install --no-audit --no-fund && npm run build
	cd crates/boxpilot-tauri && cargo build --release --locked

override_dh_auto_test:
	# Heavy: full workspace tests run pre-PR, not at package build.
	# Re-enable once the build host is provisioned with all sing-box
	# integration deps.

override_dh_auto_install:
	install -D -m 0755 target/release/boxpilotd \
	    debian/boxpilot/usr/lib/boxpilot/boxpilotd
	install -D -m 0755 target/release/boxpilot \
	    debian/boxpilot/usr/lib/boxpilot/boxpilot
	# debhelper handles the rest via debian/boxpilot.install
```

Rationale for `--locked`: matches the existing CI hygiene posture — fail
loudly if `Cargo.lock` is out of date instead of silently bumping deps at
package build time.

The `npm install` call uses `--no-audit --no-fund` because audit warnings
during package build are noise and `npm fund` can't be useful in a
non-interactive context.

## 6. Maintainer scripts

All scripts use `set -e` and respect dpkg's `$1` action codes
(`configure`, `remove`, `purge`, `upgrade`, etc.). They use idempotent
operations only — the package can be re-installed without first removing.

### 6.1 `postinst` (action: `configure`)

```sh
#!/bin/sh
set -e

case "$1" in
    configure)
        # Reload D-Bus so the new system-service file is picked up
        # without requiring a reboot. We deliberately reload, not
        # restart, because restarting dbus terminates the user session.
        if [ -d /run/systemd/system ]; then
            systemctl reload dbus.service 2>/dev/null \
                || systemctl restart dbus.service 2>/dev/null \
                || true
        fi

        # Reload systemd so the new boxpilotd.service unit is visible.
        if [ -d /run/systemd/system ]; then
            systemctl daemon-reload || true
        fi

        # Reload polkit so the new actions / rules file takes effect.
        if [ -d /run/systemd/system ] \
            && systemctl is-enabled polkit.service >/dev/null 2>&1; then
            systemctl reload polkit.service 2>/dev/null \
                || systemctl restart polkit.service 2>/dev/null \
                || true
        fi

        # Refresh icon + desktop caches.
        if which gtk-update-icon-cache >/dev/null 2>&1; then
            gtk-update-icon-cache -f -t /usr/share/icons/hicolor || true
        fi
        if which update-desktop-database >/dev/null 2>&1; then
            update-desktop-database /usr/share/applications || true
        fi
        ;;
esac

#DEBHELPER#
```

### 6.2 `prerm` (actions: `remove`, `upgrade`)

```sh
#!/bin/sh
set -e

case "$1" in
    remove|upgrade|deconfigure)
        # On *remove*, the user-managed sing-box service the daemon
        # generated will outlive the package; it is fine to leave it
        # running because it points to a copy of the (still installed)
        # config under /etc/boxpilot/active. On *purge*, postrm cleans
        # both the unit and the config.
        #
        # We do *not* stop boxpilot-sing-box.service here — pulling it
        # down on package upgrade would be a surprising regression for
        # users who depend on the proxy at upgrade time. boxpilotd itself
        # is D-Bus-activated and will be restarted on next call by the
        # new binary.
        :
        ;;
esac

#DEBHELPER#
```

### 6.3 `postrm` (actions: `remove`, `purge`)

```sh
#!/bin/sh
set -e

case "$1" in
    purge)
        # Clean runtime + config trees. Does NOT touch any external
        # /etc/systemd/system/sing-box.service — that is owned by the
        # user (see plan #6).
        rm -f  /etc/systemd/system/boxpilot-sing-box.service
        rm -rf /etc/boxpilot
        rm -rf /var/lib/boxpilot
        rm -rf /var/cache/boxpilot
        if [ -d /run/systemd/system ]; then
            systemctl daemon-reload || true
        fi
        ;;
    remove)
        # Refresh caches. User data preserved.
        if which gtk-update-icon-cache >/dev/null 2>&1; then
            gtk-update-icon-cache -f -t /usr/share/icons/hicolor || true
        fi
        if which update-desktop-database >/dev/null 2>&1; then
            update-desktop-database /usr/share/applications || true
        fi
        if [ -d /run/systemd/system ]; then
            systemctl reload dbus.service 2>/dev/null \
                || systemctl restart dbus.service 2>/dev/null \
                || true
            systemctl daemon-reload || true
        fi
        ;;
esac

#DEBHELPER#
```

## 7. `boxpilotd.service` (new file)

```ini
[Unit]
Description=BoxPilot privileged helper (D-Bus activated)
Documentation=https://github.com/BeyondCortex/BoxPilot
After=dbus.service
Requires=dbus.service

[Service]
Type=dbus
BusName=app.boxpilot.Helper
ExecStart=/usr/lib/boxpilot/boxpilotd
User=root
# Helper terminates itself on idle (see boxpilotd::idle); systemd
# restarts it on the next D-Bus call.
Restart=on-failure
RestartSec=2s
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
PrivateTmp=true
ReadWritePaths=/etc/boxpilot /var/lib/boxpilot /var/cache/boxpilot /etc/systemd/system /run/systemd /run/dbus
CapabilityBoundingSet=CAP_SYS_ADMIN CAP_NET_ADMIN CAP_DAC_OVERRIDE CAP_CHOWN CAP_FOWNER

[Install]
# No WantedBy — D-Bus activates this on demand.
```

Rationale on the sandbox profile: this is the **helper** unit, not the
managed sing-box unit. The helper has to write to multiple system
locations (the runtime-generated `boxpilot-sing-box.service`, the
release tree under `/var/lib/boxpilot`, profile bundles under
`/etc/boxpilot`), reload systemd, and talk to dbus, so its
`ReadWritePaths=` is broader than the sing-box unit's `ReadWritePaths=
/etc/boxpilot/active` from §7.1 of the linux design spec. It still keeps
`ProtectSystem=strict`, `ProtectHome=true`, `PrivateTmp=true` because
none of the helper's writes touch home or `/usr`.

## 8. `app.boxpilot.desktop`

```ini
[Desktop Entry]
Type=Application
Name=BoxPilot
GenericName=Sing-box Control Panel
Comment=Manage system sing-box from the desktop
Exec=boxpilot
Icon=boxpilot
Terminal=false
Categories=Network;
Keywords=sing-box;proxy;vpn;tun;
StartupWMClass=BoxPilot
StartupNotify=true
```

Notes:

- `StartupWMClass=BoxPilot` matches the Tauri window title from
  `tauri.conf.json` so GNOME Shell groups the window with its launcher
  icon instead of showing a generic Wayland fallback.
- `Categories=Network;` is the single main category. `desktop-file-validate`
  emits a hint for entries that list multiple main categories (the entry
  may appear more than once in the menu); we pick the most accurate one
  and rely on `Keywords=` for searchability.

## 9. Icon strategy

The repo currently ships a single 32×32 PNG at
`crates/boxpilot-tauri/icons/icon.png`. The package needs at least three
sizes for icon-cache lookup to work cleanly across DE / scaling factors.

Plan #9 generates the missing sizes with one of:

1. `convert` (ImageMagick), if available on the dev host;
2. checked-in pre-generated PNGs.

Decision: **check in pre-generated PNGs**, because relying on
ImageMagick at build time would push another build-dep onto users who
build from source. The 32 px master image is upscaled to 128 / 256 with
nearest-neighbor (it is a simple flat icon; we accept the visual hit and
will replace with a vector master in a follow-up).

Files under `packaging/linux/icons/hicolor/<N>x<N>/apps/boxpilot.png` are
the source of truth; `debian/boxpilot.install` maps them into
`/usr/share/icons/hicolor/<N>x<N>/apps/`.

## 10. `Cargo.lock` posture

The current repo commits `Cargo.lock`. `dpkg-buildpackage` will use it
as-is via `--locked`. We do **not** run `cargo update` at package build
time; users who want a fresher lock run `cargo update` and rebuild.

## 11. Upgrade semantics

| Action | Effect on user state |
|--------|----------------------|
| `apt install ./new.deb` over an existing install | Binaries replaced. `boxpilot-sing-box.service` keeps running. `boxpilotd` is D-Bus activated, so the next call exec's the new binary. Profiles, releases, diagnostics untouched. |
| `apt remove boxpilot` | Binaries + helper + desktop / icon / dbus / polkit files gone. `/etc/boxpilot`, `/var/lib/boxpilot`, `/var/cache/boxpilot` retained. `boxpilot-sing-box.service` retained — the unit references `/usr/lib/boxpilot/...` paths only via the running config under `/etc/boxpilot/active`, and the active config still works as long as the user has not removed the sing-box binary it points at. |
| `apt purge boxpilot` | Everything in `/etc/boxpilot`, `/var/lib/boxpilot`, `/var/cache/boxpilot` removed. The runtime-generated `boxpilot-sing-box.service` is removed. `boxpilotd.service` is gone with the package. External `sing-box.service` (if any) is **not** touched. |

## 12. Acceptance criteria

Plan #9 ships when:

- `dpkg-buildpackage -b -uc -us` from a clean checkout produces
  `boxpilot_0.1.0_amd64.deb` (or arch-equivalent) with no build errors.
- `lintian boxpilot_0.1.0_amd64.deb` reports no errors and no warnings
  beyond the documented exemptions in `debian/source/lintian-overrides`
  (none expected at v0.1.0; the PR records any that appear).
- `sudo apt install ./boxpilot_0.1.0_amd64.deb` installs cleanly on a
  fresh Ubuntu 24.04 / Debian 13 VM, the desktop entry shows up in the
  application menu, clicking it opens the GUI, and the GUI reaches
  `service.status` round-trip without manual D-Bus reload.
- `sudo apt remove boxpilot` leaves `/etc/boxpilot` and
  `/var/lib/boxpilot` intact.
- `sudo apt purge boxpilot` removes all three trees and the
  runtime-generated sing-box unit (if present).
- The GUI smoke procedure from plan #7 still passes against the
  installed package (all three tabs functional, profile activation
  end-to-end works).
- `cargo fmt --all -- --check`, `cargo clippy --workspace --all-targets
  -- -D warnings`, `cargo test --workspace`, `cd frontend && npm run
  build` all pass.
- README's quick start uses `apt install ./*.deb`. The dev `make
  install-helper` flow is documented under "Building from source".
- `app.boxpilot.Helper.service` has `SystemdService=boxpilotd.service`
  and the `Plan #9 will reintroduce` TODO comment is removed.

## 13. Risk register

| Risk | Mitigation |
|------|------------|
| Tauri's webkit2gtk runtime dep changes between Ubuntu 22.04 (`libwebkit2gtk-4.0`) and 24.04+ (`4.1`) | `Depends:` lists `libwebkit2gtk-4.1-0 \| libwebkit2gtk-4.0-37` so the package installs on both. Test target is 24.04; older distros are best-effort. |
| `polkit.service` is not a thing on every distro (some still use `polkit-1.service` or no service at all) | postinst's polkit reload is wrapped in `is-enabled` + `\|\| true`. Reloading polkit is best-effort. |
| systemd `daemon-reload` failure during package install on a non-systemd container (CI / docker) | Wrap in `[ -d /run/systemd/system ]` test. Same posture as Debian's own packages. |
| `dpkg-buildpackage` runs as a non-root user but the maintainer scripts assume root | dpkg always invokes maintainer scripts as root at install time; build-time scripts (rules) run as the build user and never need root. No change needed. |
| User has a custom `sing-box.service` that conflicts with `boxpilot-sing-box.service` | Plan #6 already handles the observation / migration story; plan #9 doesn't change it. The package never installs `sing-box.service` itself. |
| Frontend `npm install` failure during `dpkg-buildpackage` because of network restrictions in build environment | We accept that package builds need network access for the first build; `--locked` on the cargo side keeps things deterministic. A future plan can introduce `npm ci` against a vendored cache. |
| Running `apt install ./boxpilot_*.deb` on a machine with the dev `make install-helper` files already present | dpkg detects file conflicts and fails the install. README warns to `make uninstall-helper` before installing the deb. |

## 14. Decision log

- **Maintainer scripts as POSIX `sh` not `bash`** — `/bin/sh` is the
  Debian convention and avoids the `bash` runtime dependency just for
  install scripts.
- **No init-system fallback** — we target systemd-only systems per spec
  §1, so the maintainer scripts gate everything behind
  `[ -d /run/systemd/system ]` and skip reloads on non-systemd hosts
  (the package still installs files; runtime is undefined on sysvinit /
  openrc).
- **Single binary package, not split** — no `boxpilot-common` /
  `boxpilot-helper` / `boxpilot-gui` split. The whole product is one
  unit; splitting adds packaging complexity for no user benefit at this
  scale.
- **No `Recommends` entries** — every dep is hard `Depends`.
  `Recommends:` would let `apt install --no-install-recommends` ship a
  broken install (e.g., no polkit agent → every action fails). We pay
  the disk cost in exchange for the install always working.
- **Architecture: `any`, not `amd64`** — keeps the source package
  arch-portable for a future `arm64` build with no source change.
